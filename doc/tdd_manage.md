<h1>测试与项目管理</h1>
<p>测试应该是贯穿项目管理始终，测试（自动化）先行，即在代码编程前就进行测试用例编制。这种测试驱动的项目管理具有以下优势：</P>
<ul>
<li>
保证代码质量，提交代码都是经过测试，具有一定质量保障。
</li>
<li>
降低团队沟通成本，测试用例就是团队内部沟通的最好语言。
<pre>
例如：需求设计人员不需要再一遍一遍教开发人员业务需求是什么样，只要把想要的输入和结果做成一个测试用例就可以让开发人员准确开发编程。
</pre>
</li>
<li>
提高开发效率，降低项目成本。
<pre>
人们经常有个认识误区，认为编写测试用例增加了工作量，降低开发效率，其中这是一个认识误区，测试增加了测试用例编写成本，但它降低了项目沟通成本，提高代码质量，降低bug修复成本，
</pre>
</li>
<li>
开发过程“透明化”
<pre>
通过自动化测试，使系统开发过程透明，可以随时知道总体、每个模块、每个testcase的开发进度、质量。
</pre>
</li>
<li>
应对需求变更的“利器”。
<pre>
在项目开发过程，需求变更是不可避免的，而需求变更如何准确的确定影响范围，进行计划调整是项目管理的重要内容，而采用测试驱动，可以精确控制变更范围和影响。
</pre>
</li>
<li>
测试是应用（特别是微服务）持续进化的保障。
<pre>
在项目上线后，任何bug和需求修改都需要进行回归测试，而没有自动化测试用例进行回归测试是很耗时事情，往往实际过程中就做简单测试就上线，这无疑是“高空走钢丝”，稍一不慎就跌入深渊。
</pre>
</li>
</ul>
